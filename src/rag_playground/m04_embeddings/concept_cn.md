## 为什么嵌入向量很重要

计算机理解数字，不理解文字。要在一堆文档中搜索"气候变化的原因"，计算机需要将查询与每个文档进行_比较_。但你如何比较两个字符串？

**嵌入向量**将文本转换为固定长度的向量（浮点数列表），语义相似的文本在向量空间中彼此靠近：

```
"气候变化"       → [0.12, -0.34, 0.56, ..., 0.78]  （768 个数字）
"全球变暖"       → [0.15, -0.31, 0.52, ..., 0.81]  （非常接近！）
"文艺复兴艺术"   → [-0.42, 0.67, -0.23, ..., -0.15]（很远！）
```

这是 RAG 的"检索"部分——没有嵌入，就无法为查询找到相关文档。

## 嵌入向量的工作原理

### 嵌入模型

嵌入模型是一个被训练用于将文本映射到向量的神经网络。训练目标：含义相似的文本应该有彼此接近的向量（高余弦相似度），而不相关的文本应该彼此远离。

**nomic-embed-text**（我们的默认模型）生成 768 维向量。它在大规模文本对语料上训练，并针对检索任务进行了优化。

### 生成嵌入向量

```
文本 → Token化 → 模型 → 向量（768 个浮点数）
```

分步说明：
1. **Token化**：将文本分解为模型能理解的 Token
2. **编码**：将 Token 传入神经网络
3. **池化**：将 Token 级别的输出合并为单个句子级向量
4. **归一化**：缩放到单位长度（用于余弦相似度）

Ollama 在内部处理 1-4 步。我们只需将文本发送到 `POST /api/embed` 就能获取向量。

### 余弦相似度

衡量嵌入相似度的标准指标：

```
cosine_sim(A, B) = (A · B) / (|A| × |B|)
```

- **1.0** = 方向相同（高度相似）
- **0.0** = 正交（不相关）
- **-1.0** = 方向相反（在现代嵌入中很少见）

对于归一化向量（单位长度），余弦相似度简化为点积。

### 批量嵌入

逐条文本嵌入速度较慢。大多数 API 支持批量嵌入——一次请求发送多条文本：

```python
# 慢：N 次独立的 API 调用
for text in texts:
    vector = await provider.embed_single(model, text)

# 快：1 次 API 调用
result = await provider.embed(model, texts)
vectors = result.vectors  # N 个向量的列表
```

## 关键概念

### 嵌入维度

| 维度 | 模型示例 | 质量 | 速度 |
|------|---------|------|------|
| 384 | all-MiniLM-L6-v2 | 好 | 非常快 |
| 768 | nomic-embed-text | 更好 | 快 |
| 1024 | bge-large-en-v1.5 | 最好 | 较慢 |

更多维度 = 更强的语义捕捉能力，但需要更多内存且搜索更慢。

### 语义相似度 vs 关键词匹配

| 查询 | 关键词匹配 | 嵌入匹配 |
|------|----------|---------|
| "汽车" vs "轿车" | 不匹配 | 高相似度 |
| "开心" vs "快乐" | 不匹配 | 高相似度 |
| "银行"（河边）vs "银行"（金融机构） | 匹配！ | 低相似度（不同含义） |

嵌入捕获的是含义，而不仅仅是字符串重叠。这是语义搜索的根本优势。

## 常见陷阱

1. **嵌入模型需要先拉取**：使用本模块前运行 `python scripts/pull_models.py`。
2. **批量限制**：Ollama 有实际批量上限。非常大的批次（>100 条文本）可能超时。
3. **维度不匹配**：如果你更换嵌入模型，必须重新嵌入所有文档。768 维向量无法与 1024 维向量比较。
4. **归一化很重要**：我们的 provider 返回归一化向量。如果你使用其他来源的原始向量，在计算余弦相似度之前要先归一化。
5. **语言支持**：nomic-embed-text 主要基于英语训练。多语言检索请考虑 bge-m3。

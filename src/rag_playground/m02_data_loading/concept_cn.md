# 模块 02：数据加载

## 你将学到

- 什么是 **Document 抽象**以及为什么 RAG 系统需要它
- 如何将**纯文本**和 **Markdown** 文件加载为 Document 格式
- 如何遍历**目录**并加载其中的所有文档
- 什么是**元数据**以及它为什么对检索质量至关重要
- **编码检测**的工作原理（以及为什么 UTF-8 并不总是够用）

## 为什么数据加载很重要

RAG 在回答关于文档的问题之前，必须先能够_读取_这些文档。这听起来很简单——`open("file.txt").read()`——但现实中的文档有多种格式、编码和结构。一个健壮的 RAG 系统需要一种统一的方式来：

1. **读取**不同类型的文件（`.txt`、`.md`、`.pdf` 等）
2. **提取元数据**——这份文档是什么？它从哪里来？何时创建的？
3. **规范化**内容为标准的 `Document` 形状，供下游组件（切分器、嵌入器）使用

`Document` 抽象是 RAG 流水线的通用货币。此模块之后的每个组件——切分、嵌入、存储、检索——都使用 `Document` 对象。

## Document 抽象

一个 Document 恰好有两个属性：

```
Document
├── page_content: str          ← 实际文本
└── metadata: dict             ← 其他一切（来源、标题、日期等）
```

这有意镜像了 [LangChain 的 Document](https://docs.langchain.com/) 接口。理解这个抽象意味着你以后可以接入任何 LangChain 兼容的流水线。但我们是自己构建它，这样你能理解每一行代码。

### 为什么元数据使用 dict？

元数据是灵活的。不同的文件类型携带不同的信息：

| 文件类型 | 典型元数据 |
|---------|----------|
| `.txt` | 文件名、路径、文件大小、修改日期 |
| `.md` | 文件名、标题（来自 heading）、frontmatter 字段（作者、日期、标签） |
| 目录 | 相对路径、父目录、文件数量 |

dict 让每种加载器可以添加自己的元数据，而无需更改 `Document` 类。

### 元数据为什么对检索很重要

当你搜索向量存储时，通常需要按元数据筛选：

- "只显示来自 `faq/` 目录的文档结果"
- "按作者筛选文档"
- "限制结果为最近 30 天的文档"

元数据筛选将在模块 05（向量存储）和模块 06（检索）中详细介绍。模块 02 专注于正确地_捕获_元数据。

## 数据加载的工作原理

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  磁盘上的    │────▶│  加载器      │────▶│  Document    │
│  文件        │     │  读取文件    │     │  对象        │
└──────────────┘     └──────────────┘     └──────────────┘
```

### 步骤 1：打开文件
Python 内置的 `open()` 处理大多数情况。我们通过先尝试 UTF-8（覆盖 95% 以上的现代文档），然后回退到 latin-1 来检测编码。

### 步骤 2：读取内容
对于纯文本，整个文件就是 `page_content`。对于 Markdown，我们将 frontmatter（`---` 分隔符之间的 YAML）与正文分开解析。

### 步骤 3：提取元数据
每种加载器创建一个元数据 dict：
- `source`：文件路径（相对于数据目录）
- `filename`：仅文件名
- `file_type`：`.txt`、`.md` 等
- 加载器特定的字段（Markdown 的 frontmatter，批量加载的目录信息）

### 步骤 4：返回 Document
加载器返回一个填充了 `page_content` 和 `metadata` 的 `Document` 对象。

## 关键概念

### 编码

文本文件并不全是 UTF-8。旧文档可能使用：
- **latin-1**（ISO-8859-1）：常见于旧版 Windows 文档
- **cp1252**：英语系统上的 Windows 默认编码
- **UTF-16**：Windows 工具有时使用

我们的加载器先尝试 UTF-8，失败时回退到 latin-1 并给出警告。

### Frontmatter

Markdown 文件通常有 YAML frontmatter：

```markdown
---
title: "我的文档"
author: "张三"
date: 2024-01-15
tags: [教程, python]
---

# 我的文档

正文从这里开始……
```

加载器在 `---` 上分割，将 YAML 块解析为元数据，保留其余部分作为 `page_content`。

### 目录加载

默认递归加载整个目录：

```
data/
├── wikipedia/
│   ├── ml.md          ← 已加载
│   └── climate.md     ← 已加载
└── faq/
    ├── dynamodb.md    ← 已加载
    └── typescript.md  ← 已加载
```

你可以通过 glob 模式（`**/*.md`）或文件扩展名进行过滤。

## 常见陷阱

1. **大文件**：将 50MB 的文件加载到内存是可行的，但下游的切分和嵌入会很慢。考虑先分割大文件。
2. **二进制文件**：我们的加载器只处理文本。PDF、图片和其他二进制格式需要专门的加载器（未来扩展）。
3. **编码错误**：如果文件有混合编码，我们回退到 latin-1 会对非 latin-1 码点产生乱码。
4. **元数据冲突**：从目录加载时，文件的元数据（frontmatter）优先于目录的元数据（路径）。

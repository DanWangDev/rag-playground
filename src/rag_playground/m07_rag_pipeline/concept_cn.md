# 模块 07：RAG 流水线

## 你将学到

- 如何组装**完整的 RAG 流水线**：加载 → 切分 → 嵌入 → 存储 → 检索 → 增强 → 生成
- 如何构建**RAG 提示模板**来约束 LLM 仅使用上下文
- 如何处理**流式** RAG 响应
- 常见的**失败模式**及如何检测

## RAG 端到端工作原理

```
1. 文档已加载（模块 02）
2. 文档已切分（模块 03）
3. 片段已嵌入（模块 04）
4. 嵌入已存入向量存储（模块 05）
5. 用户提问
6. 问题嵌入 → 搜索存储 → 检索 top-k 片段（模块 06）
7. 检索的片段 + 问题 → 格式化提示
8. 提示 → LLM → 带引用的答案（模块 01）
```

### 增强步骤

这是 RAG 与普通 LLM 对话的区别所在。检索到的片段被格式化到提示中：

```
System：你是一个乐于助人的助手。仅使用以下提供的上下文回答问题。
        如果答案不在上下文中，请说"我没有足够的信息来回答这个问题"。

上下文：
[片段 1] 机器学习是人工智能的一个领域……
[片段 2] 监督学习使用标注数据……
[片段 3] 深度学习使用神经网络……

问题：什么是机器学习？

回答：
```

### 为什么系统提示词至关重要

如果不指示模型仅使用上下文，LLM 可能：
- 忽略检索到的片段，使用其训练数据
- 自信地回答文档中不存在的问题
- 编造来源

强力的 RAG 系统提示词对于获得有据可查的答案至关重要。

### 流式 RAG

检索步骤很快（约 10ms）。生成步骤较慢（数秒）。
流式输出在生成 Token 时即发送，让答案实时构建呈现。

## 失败模式

| 失败 | 症状 | 修复方法 |
|------|------|---------|
| 不相关检索 | 答案未针对问题 | 更好的切分策略、重排序 |
| 上下文溢出 | LLM 忽略部分上下文 | 更小的片段、摘要 |
| 幻觉 | 看似合理但错误的答案 | 更严格的系统提示词、引用标注 |
| 领域外问题 | LLM 回答了文档中没有的内容 | 系统提示词："说我不知道" |
